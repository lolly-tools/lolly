# Tauri Mobile Shell

iOS and Android distribution via Tauri 2. Shares the web shell source
(`shells/web/src/`) and the filesystem `state.js` bridge override with the desktop
shell; differs in capabilities, identifier, and the platform-specific `gen/`
projects generated by Tauri's mobile init commands.

## Feature subset

The mobile shell provides the capabilities `network`, `clipboard`, `wasm`,
`compose`, and `filesystem` (the last via `tauri-plugin-fs`; see
`bridge-overrides/capabilities-provided.js`). Tools that require `ffmpeg`
(sidecar transcoding) or `capture` (native headless-Chrome page capture,
desktop-only) are filtered out of the gallery — there is no `tauri-plugin-shell`
and no capture path on mobile. Camera input works through the WebView's
`getUserMedia` (an OS-permission-gated API, independent of Lolly's capability
gating).

## What's different from the desktop shell

| Concern | Desktop | Mobile |
|---|---|---|
| Identifier | `tools.lolly.app` | `tools.lolly.app.mobile` |
| Shell plugin (sidecars) | yes | no |
| Page capture (`capture`) | yes (headless Chrome) | no |
| Filesystem state | yes | yes |
| Camera | no | via WebView `getUserMedia` |
| Dev port | 5173 | 5174 |
| Platform targets | macOS / Windows / Linux | iOS / Android |

State persistence is the same on both Tauri shells: filesystem via
`tauri-plugin-fs`, at `$APPDATA/Lolly/saved-state/*.json`. The mobile override is
kept as a separate file (`bridge-overrides/state.js`) so mobile-specific storage
(scoped storage on Android, iCloud on iOS) can diverge later without touching
desktop.

## Structure

```
shells/tauri-mobile/
├── package.json              # npm scripts + JS dependencies
├── vite.config.js            # builds web shell src; overrides state.js + capabilities-provided.js (port 5174)
├── bridge-overrides/
│   ├── state.js              # filesystem state — replaces IndexedDB at build time
│   └── capabilities-provided.js  # web set + 'filesystem' (no 'capture')
├── dist/                     # Vite output consumed by Tauri (gitignored)
└── src-tauri/
    ├── Cargo.toml
    ├── build.rs
    ├── tauri.conf.json       # productName "Lolly", identifier, devUrl, bundle.iOS
    ├── capabilities/
    │   └── default.json      # fs + http permissions (no shell plugin)
    ├── icons/                # committed app icons (android/ + ios/)
    ├── gen/                  # generated by tauri android/ios init (gitignored)
    └── src/
        ├── main.rs
        └── lib.rs
```

`src-tauri/gen/` (the generated Gradle / Xcode projects) is gitignored via
`**/src-tauri/gen/`, so it is regenerated per checkout with the init command
below — it is never committed.

## First-time setup

The mobile shell has its own `node_modules` and Tauri CLI; install them first.
Platform init is a one-time step that generates `src-tauri/gen/android/` or
`src-tauri/gen/apple/`:

```bash
cd shells/tauri-mobile
npm install

# Android
npm run tauri android init

# iOS (macOS only — needs full Xcode + CocoaPods)
npm run tauri ios init
```

Icons are already committed under `src-tauri/icons/` (the `android/` and `ios/`
sets); init wires them into the native projects. To regenerate the whole set from
a source PNG: `npx @tauri-apps/cli icon path/to/icon-1024.png`.

## Dev

```bash
# Android (emulator or connected device with USB debugging)
npm run dev:android

# iOS simulator (macOS only)
npm run dev:ios
```

Both serve the frontend from the Vite dev server on port 5174 (via
`beforeDevCommand`), so hot reload works the same as web and desktop. Target a
specific iOS simulator with `npm run dev:ios -- "iPhone 15 Pro"`.

## Build

```bash
npm run build:android   # outputs APK + AAB under src-tauri/gen/android/
npm run build:ios       # outputs .ipa under src-tauri/gen/apple/
```

Each runs `vite build` (bundling `tools/` and `catalog/` into `dist/`) before the
native build.

### Signing (release)

Set credentials via environment variables — none are committed to the repo.

- **Android:** `ANDROID_KEY_STORE`, `ANDROID_KEY_STORE_PASSWORD`,
  `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`.
- **iOS:** `APPLE_DEVELOPMENT_TEAM` (Team ID), plus App Store Connect API vars
  (`APPLE_API_ISSUER` / `APPLE_API_KEY` / `APPLE_API_KEY_PATH`) for
  non-interactive signing.

## Prerequisites

- **Android:** Android Studio (SDK Platform API 33+, NDK 26+, command-line
  tools), `ANDROID_HOME` / `NDK_HOME`, and the Android Rust targets.
- **iOS:** macOS with **full Xcode** (not just Command Line Tools) and
  **CocoaPods**, plus the iOS Rust targets (`aarch64-apple-ios`,
  `aarch64-apple-ios-sim`, `x86_64-apple-ios`).

> **Note — iOS does not build on the current dev machine yet.** This machine has
> only the Xcode **Command Line Tools** (`xcode-select -p` ->
> `/Library/Developer/CommandLineTools`); `xcodebuild` and the Simulator
> (`simctl`) are absent, and `src-tauri/gen/apple/` has not been initialized.
> CocoaPods (`pod` 1.16.2), the iOS Rust targets, and the mobile `node_modules`
> are already installed. `tauri ios init` / `dev:ios` / `build:ios` therefore fail
> here until **full Xcode** is installed. The desktop `.app`/`.dmg` build is
> unaffected (it needs only Command Line Tools).

See `docs/ios-build.md` for the full iOS prerequisites, init, simulator loop,
permissions, and signing walkthrough, and `docs/build-guide.md` for the
cross-platform build reference (CLI, desktop, Android).
